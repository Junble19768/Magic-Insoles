# ADR-0001: BLE/LTE 双链路独立，手机不做数据中转

- **状态**：已采纳
- **日期**：2026-06

## 背景

儿童步态鞋垫需要同时支持：
1. **近程实时可视化**（家长用手机看足底热力图、COP 轨迹）——要求低延迟（<50ms）
2. **云端存储与 AI 分析**（生成每日健康日报）——要求可靠上传、后端特征提取与 LLM 调用

最初讨论中曾考虑「手机作为 BLE 网关，转发数据到云端」，但会增加手机端复杂度、耗电、且两条链路耦合。

## 决策

采用 **STM32 两条独立输出链路**，手机 Web 为**纯消费端**：

```
STM32
 ├── BLE Notify（50Hz）──→ 手机 Web（模式 A：近程实时可视化）
 └── LTE HTTP POST（批量）──→ 公网云后端（存储 + AI 分析）
                                    ↑ HTTPS GET
                             手机 Web（模式 B：远程查看）
```

- BLE 链路：仅用于手机近程实时渲染，数据不上传后端
- LTE 链路：STM32 直传后端，手机不参与转发
- 两种模式可同时使用或独立使用

## 备选方案

- **方案 A：手机 BLE 网关转发** — 未选原因：增加手机端后台上传逻辑、耗电、BLE 与 LTE 耦合、演示时手机必须在线才能入库
- **方案 B：仅 BLE，手机负责全部上传** — 未选原因：远程查看模式依赖手机在场，无法独立展示历史报告
- **方案 C：仅 LTE，放弃近程实时** — 未选原因：50Hz 热力图需要低延迟，LTE 批量上传无法满足实时可视化

## 影响

- **正面**：架构清晰、职责分离；近程与远程可独立演示；手机端代码简化（无上传队列）
- **负面 / 代价**：STM32 需同时维护 BLE 与 LTE 两套输出；固件 config 需写入 API Key 与后端 URL
- **后续**：固件 LTE 模块抽象 AT 指令层；API Key 与前端 `VITE_API_KEY` 保持一致
