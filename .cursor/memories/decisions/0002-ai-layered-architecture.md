# ADR-0002: AI 分层——TinyML 边缘 + LLM 云端

- **状态**：已采纳
- **日期**：2026-06

## 背景

赛题强调「AI 赋能」，项目需要在边缘与云端都体现 AI 能力。若全部放在云端：实时步态分类延迟高、依赖网络；若全部放在边缘：无法生成自然语言家长日报，且 STM32 算力有限。

TALK_WITH_AI 评审指出：「采集很多数据 + 做大模型分析」若缺乏与指导价值的关联，评委只会问「所以呢？」。需要明确各层 AI 的职责。

## 决策

采用 **两层 AI 分工**：

| 层级 | 运行位置 | 负责内容 | 输出 |
|------|---------|----------|------|
| TinyML（边缘） | STM32 | 步态分类：正常 / 内八 / 外八 | 分类结果 + 置信度 → BLE 帧 + LED/马达反馈 |
| LLM（云端） | DeepSeek API | 自然语言健康日报生成 | 150 字以内家长日报 |

- TinyML 负责**实时性**与**即时反馈**（连续 5 帧异常触发 LED/马达）
- LLM 负责**解释性**与**家长可读性**（汇总当日特征，生成建议与鼓励语）
- 不做本地大模型部署；不做云端实时推理替代 TinyML

## 备选方案

- **方案 A：仅云端 AI** — 未选原因：实时分类需上传再推理，延迟与流量不可接受；无法驱动板载即时反馈
- **方案 B：仅边缘 TinyML + 规则模板报告** — 未选原因：固定模板缺乏「AI 赋能」展示效果，家长日报质量差
- **方案 C：手机端运行小模型** — 未选原因：Web 端算力与功耗限制；与「边缘在鞋垫」叙事不符

## 影响

- **正面**：职责清晰，答辩时可分「边缘智能」与「云端语义增强」两层叙述；符合 AIoT 闭环叙事
- **负面 / 代价**：需采集数据训练 TinyML（可简单过拟合演示）；依赖 DeepSeek API 可用性与密钥管理
- **后续**：TinyML 模型待硬件组采集步态数据后训练；Prompt 模板见 `techContext.md`；异常比例等特征由 `services/feature.py` 汇总后送入 LLM
