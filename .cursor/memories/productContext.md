# Product Context（为什么与设计理念）

> 记录项目存在的理由、要解决的问题，以及讨论中沉淀的设计理念。
> 最近更新：2026-07-06

## 要解决的问题

儿童期（4–12 岁）是足弓与下肢力线的发育黄金期。内八/外八步态若未及时纠正，长期将导致膝关节、踝关节及脊柱的不可逆损伤。传统矫治依赖医院定期复诊，缺乏居家、实时、量化的连续监测手段。

## 竞品差距（来自 TALK_WITH_AI 评审结论）

手机、手表、运动手环已能做步数/步频/GPS 轨迹（约 80%），但**无足底压力分析能力**。本项目差异化在于：

- **步态异常筛查**：内八/外八检测 + COP 轨迹 + 足底压力热力图
- **运动发育评估**：平衡能力（单脚站立 30s 评估）、协调能力、左右脚差异
- **专业指标**：COP（压力中心）轨迹是专业步态分析常用指标，接近运动科学/医学评估

> 评委核心追问：「所以呢？」—— 检测结果必须与教育/运动指导价值关联，不能只做数据采集 + 图表展示。

## 目标用户与场景

| 用户 | 场景 | 模式 |
|------|------|------|
| 家长（PC） | 远程监控 Dashboard、运动数据、步态分析、GPS、AI 报告 | 模式 B：HTTPS 拉取后端 |
| 家长（手机） | 近程 BLE 实时热力图 + 平衡评估 | 模式 A：Web Bluetooth |
| 儿童 | 穿戴鞋垫日常活动，异常时 LED/马达提醒 | 被动使用 |

## 产品形态

- **PC 端**：远程监控后台（Dashboard、GPS、运动数据、步态分析、AI 报告），东亚健康软件后台视觉风格
- **手机端**：上述页面响应式适配 + BLE 近程功能（设备管理、实时压力可视化、平衡评估）

## 设计理念与原则

1. **比赛从简原则**：人手不足，功能从简，保证核心演示闭环；当前阶段只需提交视频
2. **AI 分层**：TinyML 负责实时性（边缘推理），LLM 负责解释性（云端日报）——职责清晰，避免「AI 硬塞」
3. **Show, don't claim**：用具体数据（COP 轨迹、异常比例、平衡评分）胜于空洞宣传语
4. **信任建立优先于规模展示**：专业指标 + 案例数据，而非「行业领先」类表述

## 视觉设计方向（前端）

- 东亚健康软件后台风格：白底 + 浅灰卡片，大量留白
- 主色调：`#163B31`（品牌绿），辅色 `#2F8F78`
- 数据卡片式布局，指标数字突出；图表色系沉稳（蓝绿渐变，避免霓虹色）
- 参考：小米健康、华为运动健康后台的数据看板风格
- 响应式断点：`<768px` 手机底部 Tab；`≥768px` PC 侧边栏；`≥1200px` 最大宽度 1400px 居中

## 明确的取舍

| 取舍项 | 决策 | 原因 |
|--------|------|------|
| 用户登录系统 | 无，单设备固定 API Key | 只有一台设备，增加复杂度无收益 |
| iOS 兼容 | 不做，仅 Android Chrome | Web Bluetooth 支持良好，演示用安卓机 |
| 离线缓存 | 暂不实现 | 嵌入式部分抽象处理 |
| 数据库 | SQLite（演示阶段） | 单设备，无需复杂数据库 |
| LLM | DeepSeek API（云端） | 有账号，负责解释性报告 |
| 标定曲线 | 待后续补充 | 当前只有标定 Demo |
| 足底坐标映射 | 待后续确定（TBD-1） | 留占位，不阻塞骨架开发 |

## 反面清单（不要做什么）

- 不做 iOS Web Bluetooth 支持
- 不做用户注册/登录系统
- 不做 PostgreSQL 等复杂数据库
- 不过度宣传「行业领先」「世界首创」
- 不把步数/步频/GPS 作为核心卖点（竞品已覆盖）
- 不混淆 COP（压力中心）与 COM（人体质心）—— 鞋垫测的是 COP

## 功能优先级

### 必须实现（比赛核心演示）

1. BLE 接收 32 路压力值 → Three.js 足底热力图实时渲染
2. COP 轨迹实时计算与展示
3. STM32 TinyML 步态分类 + 置信度显示
4. LED/震动马达异常实时反馈
5. STM32 经 LTE 直传原始压力帧至后端
6. 后端调用 DeepSeek API 生成每日健康日报
7. 前端远程查看 AI 日报

### 扩展实现（时间充裕时）

- GPS 运动轨迹地图
- 历史步态评分趋势图
- 维稳/平衡测试标准化模式（`/balance` 页已实现骨架）
- OLED 显示设备状态
