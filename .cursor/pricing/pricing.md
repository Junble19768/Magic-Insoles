# Cursor 模型定价与收费指数说明

## 推荐选型（Agent 重度 · 订阅制）

面向 **Agent 重度使用**、**优先烧 Auto 订阅池** 的极简 API 选型。原则：成本优先、按 Tier 逐级升级，仅在上一级搞不定时才升档。

### 常备模型

| 档位 | 模型 | Agent 相对成本 | 用途 |
|------|------|---------------:|------|
| **订阅默认** | **Auto** | 499 | 独立 Auto 池，不占 API 计费；能走池子的任务优先用它 |
| **Tier 2** | **Grok 4.3** | 233 | 日常 Chat、CRUD、组件拼装 |
| **Tier 2 备选** | Sonnet 4.5 / Kimi K2.5 | 1225 / 247 | Grok 搞不定时的 fallback；国内项目优先 Kimi |
| **Tier 3** | **Composer 2.5** | 215 | Cursor Agent 主力：多文件重构、修 Bug 循环 |
| **Tier 3 备选** | **Grok Build 0.1** | 189 | 自动化流水线、CI、非 IDE 的 agent 场景 |
| **Tier 4** | **Opus 4.8** | 2655 | 深度逻辑（Claude 侧）：算法、架构、隐蔽 Bug |
| **Tier 4** | **GPT-5.5** | 3128 | 深度逻辑（OpenAI 侧）：与 Opus 4.8 二选一 |

### 升级路径

```
Auto（订阅池，默认路由）
  ↓ 池用尽 / 需要可控模型
Tier 2  Grok 4.3 ──fallback──→ Sonnet 4.5 / Kimi K2.5
  ↓ Chat 搞不定 / 跨文件 Agent
Tier 3  Composer 2.5（IDE 内） / Grok Build 0.1（流水线）
  ↓ 深度推理 / 普通模型死循环
Tier 4  Opus 4.8 ↔ GPT-5.5
```

### 关键说明

**Auto 不是 Tier 1。** Auto 从 Composer 2.5 用量池自动路由，面向 Agent/Chat，与 Tier 3 有重叠，只是计费走独立订阅池。Tier 1（Tab 补全、Lint 修复）由 Cursor 内置模型处理，无需手动选型。

**Tier 3 分工：** Composer 2.5 负责 Cursor 内多文件 Apply 与报错循环；Grok Build 0.1 负责 CI/自动化等 IDE 外场景。两者 Agent 指数接近（215 vs 189），成本差异不大。

**Tier 4 不建议常备 Fable 5。** Fable 5 相对成本 4085，比 Opus 4.8 贵约 54%，且不在 `MODEL.md` Tier 4 名单；除非实测明显更强，否则 Opus 4.8 + GPT-5.5 已足够。

**Tier 2 不宜单点。** Grok 4.3 省钱但复杂框架一致性弱于 Sonnet；建议至少保留 Sonnet 4.5 或 Kimi K2.5 作备选，不必日常常驻。

## 一、为什么需要收费指数

Cursor 定价包含四种 Token 类型，日常消耗结构因场景而异：

| Token 类型 | 典型场景 | 日常占比（粗估） |
|-----------|---------|----------------|
| 输入 | 首轮上下文、新文件 | 10–20% |
| 缓存写入 | Anthropic 首轮写缓存 | 5–10% |
| 缓存读取 | Agent 多轮复用上下文 | **45–55%** |
| 输出 | 代码、解释、推理 token | **35–42%** |

仅按「输入单价」排序会严重低估 Opus、GPT-5.5 等模型的真实账单。收费指数通过加权合成，将四种单价合并为一个可排序的数字。

## 二、三种场景指数

### 2.1 Agent 指数（默认主排序列）

**适用场景**：Agent 自动修 Bug、多文件重构、工具调用循环——Cursor 最耗 quota 的用法。本仓库默认按 **Agent 重度** 偏好校准权重。

**有缓存写入**（Anthropic 系列）：

```
Agent指数 = 0.08 × 输入 + 0.05 × 缓存写入 + 0.45 × 缓存读取 + 0.42 × 输出
```

**无缓存写入**（Composer、GPT、Gemini 等）：

```
Agent指数 = 0.13 × 输入 + 0.45 × 缓存读取 + 0.42 × 输出
```

无缓存写入时，原「缓存写入」权重（0.05）并入「输入」（0.08 + 0.05 = 0.13）。

**Tier 4 修正系数**：

Thinking 模型（Opus 系列、GPT-5.5）的实际输出 token 通常高于公式假设，因此额外乘以 **1.3**：

```
Tier 4 最终 Agent指数 = 上述公式结果 × 1.3
```

| 权重项 | Agent 重度 | 设计依据 |
|-------|-----------|---------|
| 输入 | 8% / 13% | Agent 首轮灌上下文后，增量输入占比下降 |
| 缓存写入 | 5% | 仅 Anthropic；首轮写缓存一次性成本 |
| 缓存读取 | **45%** | 多轮 Agent 循环中，复用上下文是最大开销 |
| 输出 | **42%** | 代码 diff、工具参数、报错分析；输出单价通常是输入的 5–15 倍 |

---

### 2.2 Tab 指数

**适用场景**：Tier 1 行内补全、单行修改——上下文长、输出极短。

**有缓存写入**：

```
Tab指数 = 0.15 × 输入 + 0.05 × 缓存写入 + 0.70 × 缓存读取 + 0.15 × 输出
```

**无缓存写入**：

```
Tab指数 = 0.85 × 输入 + 0.70 × 缓存读取 + 0.15 × 输出
```

| 权重项 | 占比 | 设计依据 |
|-------|-----|---------|
| 输入 | 15% / 85% | Tab 每次触发上下文相对固定；无缓存时输入即主要成本 |
| 缓存读取 | 70% | 同一文件反复补全时，缓存命中极高 |
| 输出 | 15% | 单行补全，输出 token 极少 |

---

### 2.3 Chat 指数

**适用场景**：Tier 2 侧边栏日常问答——输入输出较均衡。

**有缓存写入**：

```
Chat指数 = 0.35 × 输入 + 0.05 × 缓存写入 + 0.20 × 缓存读取 + 0.40 × 输出
```

**无缓存写入**：

```
Chat指数 = 0.40 × 输入 + 0.20 × 缓存读取 + 0.40 × 输出
```

| 权重项 | 占比 | 设计依据 |
|-------|-----|---------|
| 输入 | 35% / 40% | 每轮对话都带入问题与部分上下文 |
| 缓存读取 | 20% | 同一会话内有缓存，但轮次少于 Agent |
| 输出 | 40% | 解释 + 代码块，输出量介于 Tab 与 Agent 之间 |

## 三、辅助列说明

| 列名 | 计算方式 | 用途 |
|------|---------|------|
| **输出倍率** | 输出 ÷ 输入 | 衡量「说多了有多贵」；Thinking 模型通常较高 |
| **相对成本** | Agent指数 ÷ 最低 Agent指数 × 100 | 以最便宜模型（GPT-5.4 Nano）为基准 100，直观倍数对比 |

**Excel 公式示例**（假设 B=输入, C=缓存写入, D=缓存读取, E=输出）：

```excel
=IF(C2="", 0.13*B2+0.45*D2+0.42*E2, 0.08*B2+0.05*C2+0.45*D2+0.42*E2)
```

Tier 4 模型可在 Agent 指数结果上再乘 `1.3`。

## 四、任务级别（Tier）说明

Tier 定义来自 [`MODEL.md`](MODEL.md)，与收费指数的关系如下：

| Tier | 名称 | 核心业务场景 | 推荐排序列 | 代表模型 |
|------|------|-------------|-----------|---------|
| **Tier 1** | 微调与高频改动 *(Micro-Edits)* | 行内 Tab 补全、单行 UT、正则/JSON 转换、Lint 修复 | **Tab 指数** | GPT-5 Mini / 5.4 Nano、Codex Mini、Gemini Flash、Haiku 4.5 |
| **Tier 2** | 标准模块开发 *(Everyday Dev)* | CRUD、组件拼装、框架 API、侧边栏 Chat 答疑 | **Chat 指数** | Sonnet 4/4.5、GPT-5.1/5.2、Kimi K2.5、GLM 5.2、Grok 4.3 |
| **Tier 3** | 跨文件复杂重构 *(Deep Refactor)* | 3+ 文件 Bug 追踪、依赖解耦、Agent 修 Bug 循环 | **Agent 指数** | **Composer 2.5**、Sonnet 4.6/5、GPT-5.4、Grok Build 0.1 |
| **Tier 4** | 盲区探索与高级架构 *(Arch & Logic)* | 底层算法、技术选型、内存泄漏/死锁/分布式事务 | **Agent 指数 × 1.3** | Opus 4.5–4.8、GPT-5.5 |
| **Auto** | 自动选型 | Cursor 从 Composer 2.5 用量池自动路由 | 同 Agent 指数 | Auto |

### 各 Tier 选型逻辑

**Tier 1 — 追求毫秒响应与极限低成本**

模型部署在高频流式通道（Tab 预测），不需要深度推理，只需语法模式匹配。Gemini Flash 虽属 Tier 1，但因大上下文常被用作「项目文档扫描器」。

**Tier 2 — 性价比与规范性的黄金平衡**

日常编码中流砥柱，上下文遵从度高、代码格式规范。国内自备 API 推荐 Kimi K2.5、GLM 5.2。

**Tier 3 — 打赢 Agent 报错循环**

需要模型频繁调用工具（读写文件、跑终端），并在报错后自动迭代修正。**Composer 2.5** 是 Cursor Agent 核心，Agent 指数仅约为 Sonnet 的 1/5–1/6，重度 Agent 场景首选。

**Tier 4 — 不计成本换深度推理**

模型输出前经思维链（CoT）慢思考，适合普通模型无法解决的底层逻辑问题。**严禁**用于 Tab 补全或日常 Chat——不仅贵，且每次响应需等待数秒思考。
